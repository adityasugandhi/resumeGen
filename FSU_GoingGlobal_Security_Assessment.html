1. Executive Summary

Overall Risk: HIGH

This assessment identified 9 security vulnerabilities (2 Critical, 3 High, 4 Medium) in the GoingGlobal institutional access authentication mechanism used by Florida State University. The findings indicate that the current authentication model poses significant risk to the confidentiality and integrity of FSU's institutional subscription.

GoingGlobal (now part of Interstride) provides career and employment resources to FSU students, faculty, and staff via an institutional subscription. Access is granted through a static URL containing an accid parameter that serves as the sole authentication credential.

Testing confirmed that this URL functions as a permanent, shared credential with no expiration, no per-user identification, no network restrictions, and no audit trail. Anyone who obtains this URL — through sharing, browser history, HTTP referrer leakage, or brute-force enumeration — gains unrestricted institutional access indefinitely.

2.2 Methodology

This assessment followed a gray-box testing approach. As an authorized FSU affiliate with legitimate access to the GoingGlobal platform, the tester analyzed the authentication mechanism used for institutional access. Testing focused exclusively on the authentication and session management layer — no attempts were made to access or modify backend databases, other institutions' data, or platform source code.

Tools & Techniques

  • Manual URL parameter analysis and manipulation
  • Browser Developer Tools (Network tab, Cookie inspector, Console)
  • Playwright browser automation for systematic testing
  • Off-network access verification (non-FSU IP)
  • Token structure reverse-engineering (pattern analysis)
  • Session persistence and cookie behavior testing

2.3 Ethical Boundaries

  • All testing performed using the tester's own legitimate FSU-provided access credentials
  • No unauthorized access to other institutions' subscriptions was attempted or achieved
  • No data exfiltration, modification, or destruction was performed
  • No denial-of-service or load testing was conducted
  • Brute-force enumeration was analyzed theoretically; no automated attacks were launched against the production system
  • This report is submitted for responsible disclosure to FSU IT Security

3. Vulnerability Details

V-001: Static Non-Expiring Authentication Token

CRITICAL · CVSS 9.1

Description

The accid URL parameter used for institutional authentication is a static, non-rotating token. Once generated, this token does not expire, is not rotated on a schedule, and cannot be revoked without vendor intervention. The token functions as a permanent master key to the institution's entire GoingGlobal subscription.

Attack Scenario

An attacker who obtains the token through any vector (link sharing, browser history, referrer leakage, brute force) gains permanent, irrevocable access to the institution's subscription. The token remains valid indefinitely, meaning a single exposure event creates a persistent security breach with no automatic remediation.

Evidence

# Token structure (redacted):

accid=<institution_code>vv<padding>vv<numeric_id>vv<padding>

# Same token tested weeks apart — still valid

# No expiration header, no token rotation observed

Impact

Unauthorized parties gain full access to FSU's paid GoingGlobal subscription content. This could result in financial loss (subscription abuse), reputational damage, and potential violation of FSU's licensing agreement with GoingGlobal/Interstride.

Remediation

  • Implement time-limited tokens (e.g., 24-hour expiration with automatic rotation)
  • Use cryptographically signed tokens (HMAC or JWT) with embedded expiration timestamps
  • Establish a token rotation schedule and revocation mechanism
CWE: CWE-798 (Hard-coded Credentials) OWASP: A07:2021 Identification and Authentication Failures

V-002: No Per-User Authentication / Authorization

CRITICAL · CVSS 9.1

Description

The authentication mechanism grants access at the institution level rather than the individual user level. There is no requirement for users to authenticate with personal credentials (e.g., FSU NetID/password). A single shared URL serves as the credential for all users at the institution.

Attack Scenario

Because there is no individual identity verification, it is impossible to distinguish between a legitimate FSU student and an unauthorized external user. Former students, expelled students, or completely unaffiliated individuals can access the platform with the same privileges as current students and faculty.

Evidence

# No login prompt, no username/password, no SSO redirect

# Accessing URL from non-FSU network:

HTTP 200 OK — Full access granted

# Session cookie set:

ip_login_no_cache=[session_value]

# No user identity embedded in session

Impact

FSU cannot enforce its acceptable use policy, cannot revoke access for individual users, and cannot comply with per-seat licensing terms if applicable. Access control is effectively non-existent.

Remediation

  • Integrate with FSU's SAML 2.0 / Shibboleth identity provider for SSO
  • Require FSU NetID authentication before granting platform access
  • Implement role-based access control (student, faculty, staff)
CWE: CWE-287 (Improper Authentication) OWASP: A01:2021 Broken Access Control

V-003: No Network / IP Binding

HIGH · CVSS 7.5

Description

The authentication URL is not restricted to FSU's campus network IP ranges. Testing confirmed that the accid token grants full access from any IP address worldwide, including residential networks, VPNs, and mobile data connections with no affiliation to FSU.

Attack Scenario

An attacker located anywhere in the world can use the URL to access FSU's subscription. There is no geographic or network-based restriction that would limit exposure if the URL is leaked. This dramatically increases the blast radius of any credential exposure event.

Evidence

# Test from non-FSU residential IP (off-campus):

$ curl -I "https://online.goinglobal.com/...?accid=[REDACTED]"

HTTP/2 200

set-cookie: ip_login_no_cache=[value]; path=/

# Full institutional access confirmed from off-campus network

# No VPN, proxy, or FSU network connection required

Impact

The lack of network binding means that token exposure leads to unrestricted global access. Even basic IP-based controls (common in library/database access) are absent, removing a standard defense-in-depth layer.

Remediation

  • Restrict accid access to FSU campus IP ranges (128.186.0.0/16, etc.)
  • Require FSU VPN for off-campus access (consistent with library database access patterns)
  • Implement geographic anomaly detection for access from unexpected locations
CWE: CWE-284 (Improper Access Control) OWASP: A01:2021 Broken Access Control

V-004: Predictable Token Structure (Enumeration Risk)

HIGH · CVSS 7.3

Description

The accid token uses a predictable structure: an institution code separated by vv delimiters, padding characters (l, i, x), and a numeric identifier. This obfuscation provides no cryptographic security and is trivially reversible.

Attack Scenario

An attacker who understands the token format could enumerate other institutions' access tokens by iterating through numeric ID ranges with known institution codes. While the institution code acts as a partial mitigation (requiring knowledge of the specific code), the numeric ID component follows a sequential pattern that reduces the search space for brute-force attacks.

Evidence

# Token structure analysis:

Format: {inst_code}vv{padding}vv{numeric_id}vv{padding}

# Observations:

- "vv" used as consistent delimiter

- Padding chars: 'l', 'i', 'x' (non-random, decorative)

- Numeric ID: Sequential integer (low entropy)

- No HMAC, no signature, no cryptographic binding

Impact

Successful enumeration could grant unauthorized access to other institutions' GoingGlobal subscriptions, turning a single-institution vulnerability into a platform-wide compromise. Even partial enumeration could expose the institution codes and ID patterns for hundreds of subscribing universities.

Remediation

  • Replace obfuscated tokens with cryptographically random UUIDs (128+ bits of entropy)
  • Implement HMAC-SHA256 signed tokens with server-side validation
  • Decouple institution identifiers from authentication tokens
CWE: CWE-330 (Insufficient Randomness) OWASP: A07:2021 Identification and Authentication Failures

V-005: Session Fixation / Cookie Persistence

MEDIUM · CVSS 5.4

Description

Once the ip_login_no_cache session cookie is set after initial authentication, it persists even when subsequent requests are made with invalid or missing accid parameters. The session does not re-validate the authentication token on each request, allowing continued access after a single successful authentication.

Attack Scenario

An attacker who briefly accesses a legitimate user's browser (e.g., on a shared or public computer) can extract the session cookie and use it from a different machine to maintain access, even if the original URL is no longer available to them. The session cookie acts as a secondary, equally valid credential.

Evidence

# Step 1: Authenticate with valid accid

Set-Cookie: ip_login_no_cache=[session_value]

# Step 2: Access with invalid accid (cookie still present)

HTTP 200 — Access maintained via cookie

# Step 3: Access with no accid parameter at all

HTTP 200 — Access maintained via cookie

# Cookie persists across browser sessions (no expiry observed)

Remediation

  • Set explicit cookie expiration (e.g., 8-hour session timeout)
  • Re-validate authentication on sensitive operations
  • Bind session cookies to client fingerprint (User-Agent + IP)
  • Implement session invalidation on the server side
CWE: CWE-384 (Session Fixation) OWASP: A07:2021 Identification and Authentication Failures

V-006: No Rate Limiting on Auth Endpoint

MEDIUM · CVSS 5.3

Description

The authentication endpoint does not implement rate limiting or throttling on requests with invalid accid parameters. This allows automated tools to rapidly test large numbers of token combinations without being blocked, locked out, or flagged.

Attack Scenario

Combined with V-004 (predictable token structure), an attacker could script automated requests cycling through numeric ID ranges to discover valid institution tokens. Without rate limiting, thousands of attempts per minute are feasible, making brute-force enumeration practical.

Remediation

  • Implement rate limiting (e.g., max 10 auth attempts per IP per minute)
  • Add exponential backoff for repeated failures from the same source
  • Deploy CAPTCHA after threshold failures
  • Alert on anomalous authentication patterns
CWE: CWE-307 (Improper Restriction of Excessive Auth Attempts) OWASP: A07:2021 Identification and Authentication Failures

V-007: URL-Based Credential Exposure

MEDIUM · CVSS 5.3

Description

The authentication credential is transmitted as a URL query parameter rather than in an HTTP header or POST body. URLs are logged in browser history, server access logs, proxy logs, browser extensions, and HTTP Referer headers. This creates multiple passive exposure vectors for the credential.

Attack Scenario

The URL containing the authentication token may be inadvertently shared through: (1) copy-pasting from browser address bar, (2) browser sync across devices, (3) browser extension telemetry, (4) HTTP Referer headers sent to third-party resources loaded on the page, (5) corporate proxy/firewall logs, (6) web archive crawlers.

Remediation

  • Transition to POST-based authentication (credentials in request body)
  • Use HTTP-only, Secure, SameSite cookies for session management
  • Set Referrer-Policy: no-referrer header
  • If URL-based auth must persist, implement one-time-use tokens that redirect to cookie-based sessions
CWE: CWE-598 (Information Exposure Through Query Strings) OWASP: A04:2021 Insecure Design

V-008: No Audit Trail / Access Logging

MEDIUM · CVSS 5.0

Description

Because authentication is performed at the institution level with no individual user identification, there is no audit trail that could identify who accessed the platform, when, or from where. This makes incident response, forensic analysis, and compliance reporting impossible.

Attack Scenario

If the authentication URL is compromised and unauthorized access occurs, FSU's IT Security team would have no way to determine: the scope of the breach, how many unauthorized users accessed the system, what data was viewed or downloaded, or when the compromise began. Incident response is effectively blind.

Remediation

  • Implement per-user authentication to enable individual access logging
  • Log all authentication events with timestamp, IP, User-Agent, and user identity
  • Establish log retention policy (minimum 90 days)
  • Set up automated alerts for anomalous access patterns
  • Integrate logs with FSU's SIEM (Security Information and Event Management) system
CWE: CWE-778 (Insufficient Logging) OWASP: A09:2021 Security Logging and Monitoring Failures

V-009: No Anti-Scraping / Automated Data Export Protection

HIGH · CVSS 7.5

Description

GoingGlobal runs on Drupal 7 (Apache/2.4.58, Ubuntu) with no Web Application Firewall (WAF), no JavaScript-based bot challenges, and no rate-limiting headers on authenticated endpoints. Once a session is established via the static accid token, the platform serves standard server-rendered HTML pages that are trivially parseable by automated tools (curl, Scrapy, Playwright, Puppeteer). The robots.txt requests a 10-second crawl delay, but this is advisory only and not enforced server-side. No X-RateLimit, Retry-After, or equivalent throttling headers are present.

Attack Scenario

An attacker who obtains the authentication URL (see V-001) writes a simple script to:

  1. Authenticate once using the static accid URL, capturing the ip_login_no_cache session cookie
  2. Crawl all country/city career guides, employer directories, and job listings systematically
  3. Extract and store the entire dataset (salary data, visa requirements, employer contacts, interview guides)
  4. Republish the scraped content as a competing free resource or sell the dataset

Because there is no WAF (Cloudflare, Akamai, AWS WAF), no CAPTCHA challenges, no browser fingerprinting, and no enforced rate limiting, a single script using requests or headless browser automation can exfiltrate the entire platform's content within hours. The Drupal 7 server-rendered pages have predictable URL patterns, making full-site scraping straightforward.

Evidence

# HTTP Response Headers (no WAF, no rate limiting):

Server: Apache/2.4.58 (Ubuntu)

X-Generator: Drupal 7 (https://www.drupal.org)

X-Drupal-Cache: HIT

X-Content-Type-Options: nosniff

Cache-Control: public, max-age=300

# Missing protections:

X-RateLimit-*: NOT PRESENT

Retry-After: NOT PRESENT

CF-Ray / Akamai / AWS WAF: NOT PRESENT

# robots.txt (advisory only, not enforced):

Crawl-delay: 10

# No server-side enforcement of crawl delay

# Content is server-rendered HTML (Drupal 7)

# No JavaScript-rendered SPA — trivially parseable

Impact

Complete data exfiltration of GoingGlobal's proprietary content database. This includes country/city career guides, salary benchmarks, employer directories, H1B visa data, interview advice, and work permit requirements — the core intellectual property that FSU (and all subscribing institutions) pay to access. An attacker could create a mirror site, undermining the vendor's business model and potentially violating FSU's licensing agreement. Because there is no per-user authentication (V-002) or audit logging (V-008), the scraping would be undetectable and unattributable.

Remediation

  • Deploy a WAF (Cloudflare, AWS WAF, or similar) with bot detection and JavaScript challenges
  • Implement server-side rate limiting: max 30 requests/minute per session, with exponential backoff
  • Add CAPTCHA challenges after detecting rapid sequential page access patterns
  • Monitor for anomalous session behavior (e.g., 100+ pages accessed in a session, systematic URL pattern traversal)
  • Implement download/export quotas per session (if export features exist)
  • Use dynamic content rendering (lazy-loaded, API-driven) to increase scraping difficulty
  • Set X-Robots-Tag: noarchive, nosnippet to prevent search engine caching of authenticated content
CWE: CWE-770 (Allocation of Resources Without Limits) OWASP: A04:2021 Insecure Design

4. Attack Scenarios

The following scenarios illustrate realistic attack vectors that exploit the identified vulnerabilities. Each scenario is mapped to the relevant vulnerability IDs and assessed for likelihood and impact.

Scenario A

Link Sharing & Social Propagation

Vector: A well-intentioned FSU student shares the GoingGlobal access URL in a group chat, Discord server, Reddit post, or email chain to help peers access career resources.

Exploitation: The URL propagates beyond the intended audience. Non-FSU users discover the link and gain full institutional access. Because the token never expires, the link remains functional indefinitely, even if the original post is deleted.

Vulnerabilities Exploited: V-001, V-002, V-003

Likelihood: Very High — This is the most probable scenario. Users routinely share helpful links.

Impact: Widespread unauthorized access with no expiration or revocation capability.

Scenario B

Browser History Harvesting

Vector: A student accesses GoingGlobal from a shared or public computer (library, computer lab, family device). The full authentication URL is stored in browser history.

Exploitation: The next user of the computer (or malware present on the machine) harvests the URL from browser history. Infostealers specifically target browser history and bookmarks as high-value credential sources.

Vulnerabilities Exploited: V-001, V-003, V-007

Likelihood: High — FSU has public computer labs; infostealer malware is increasingly common.

Impact: Silent credential theft with no detection mechanism (V-008).

Scenario C

HTTP Referrer Header Leakage

Vector: The GoingGlobal page loads third-party resources (analytics scripts, embedded content, advertisements). The browser automatically sends the full page URL, including the accid parameter, in the HTTP Referer header to those third parties.

Exploitation: Third-party services receive and log the full authentication URL. This data could be accessed by employees of those services, exposed in data breaches, or sold as part of analytics datasets.

Vulnerabilities Exploited: V-001, V-007

Likelihood: High — Most websites load third-party resources; referrer leakage is automatic and silent.

Impact: Passive, undetectable credential exposure to multiple third parties.

Scenario D

Brute-Force Institution Enumeration

Vector: An attacker who has obtained one institution's token (e.g., FSU's) reverse-engineers the token structure and systematically tests variations to discover other institutions' tokens.

Exploitation: Using the predictable {code}vv{padding}vv{id}vv{padding} format, the attacker iterates through numeric IDs combined with known university codes (publicly available institution lists). Without rate limiting (V-006), automated testing can proceed at high speed.

Vulnerabilities Exploited: V-004, V-006

Likelihood: Medium — Requires some technical skill and knowledge of the token format, but the coupling of code+ID provides partial mitigation.

Impact: Platform-wide compromise affecting multiple subscribing institutions.

Scenario E

Session Cookie Hijacking

Vector: An attacker on a shared network (campus WiFi, coffee shop) intercepts or steals the ip_login_no_cache session cookie through XSS, network sniffing (if any HTTP requests exist), or physical access to an unlocked browser.

Exploitation: The stolen cookie is replayed from a different machine. Due to session persistence without re-validation (V-005), the attacker gains sustained access without needing the original URL.

Vulnerabilities Exploited: V-005, V-008

Likelihood: Medium — Requires proximity or malware; mitigated somewhat by HTTPS.

Impact: Unauthorized access that persists independently of the original authentication URL.

Scenario F

Web Archive / Search Engine Indexing

Vector: The authentication URL is posted on a publicly accessible webpage (university help page, forum, knowledge base article). Search engine crawlers index the URL, or the Wayback Machine archives the page.

Exploitation: Anyone searching for GoingGlobal institutional access URLs discovers the credential via a simple Google search (Google dorking). Even if the original page is updated, cached/archived versions retain the URL.

Vulnerabilities Exploited: V-001, V-002, V-007

Likelihood: Medium — Depends on where the URL is published; university resource pages are common vectors.

Impact: Permanent, publicly discoverable credential exposure.

Scenario G

Insider Threat / Departing Employee

Vector: An FSU employee, student worker, or graduate assistant who had legitimate access to the GoingGlobal URL bookmarks it or saves it before leaving the university.

Exploitation: After the individual's FSU affiliation ends (graduation, termination, departure), they retain permanent access via the bookmarked URL. Because there is no per-user authentication (V-002), there is no account to deactivate in the offboarding process.

Vulnerabilities Exploited: V-001, V-002, V-003

Likelihood: Very High — Every departing student/employee who ever used GoingGlobal retains access.

Impact: Cumulative unauthorized access that grows with each graduation cycle.

Scenario H

Automated Data Scraping & Mass Export

Vector: An attacker authenticates once using the static accid URL, captures the session cookie, then deploys an automated scraping script to systematically download the entire GoingGlobal content library.

Exploitation: The platform runs on Drupal 7 with server-rendered HTML, no WAF, no JavaScript bot challenges, and no enforced rate limiting. A simple Python script using requests + BeautifulSoup (or headless Playwright) can:

  • Enumerate all country/city guide URLs via predictable Drupal path patterns
  • Download salary data, employer directories, visa guides, and interview resources
  • Export everything to structured JSON/CSV for redistribution
  • Create a self-hosted mirror or API endpoint serving the scraped data

The robots.txt specifies a 10-second crawl delay, but this is purely advisory — the server does not enforce it. Even respecting the delay, a scraper could exfiltrate the entire database within 24–48 hours. Without the delay, the process takes under an hour.

Vulnerabilities Exploited: V-001, V-005, V-006, V-008, V-009

Likelihood: High — Low technical barrier; automated scraping tools are readily available; Drupal 7 HTML is trivially parseable.

Impact: Complete exfiltration of GoingGlobal's proprietary content. The attacker could redistribute the data freely, create a competing service, or sell the dataset — undermining the vendor's revenue and potentially triggering licensing violations for FSU.
